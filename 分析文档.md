# 分析文档

## 1. 项目定位

这个项目是一个 **Claude Code 多模型启动器**，核心目标不是实现 AI 能力本身，而是为已经安装好的 Claude Code 提供一个“模型切换入口”。

它的作用可以概括为三件事：

1. 管理多个模型配置文件；
2. 让你通过 `cc` 或 `ccs` 命令快速选择模型；
3. 以不同的 `--settings` 配置启动 Claude Code，从而让不同终端窗口可以独立使用不同模型。

从 README 可以看到，项目主打的能力是“一个命令切换不同 AI 模型，各窗口独立运行，互不影响”。这意味着它本质上是一个 **启动器 / 包装器脚本项目**，而不是一个长期驻留后台的程序，也不是一个会自动同步整个项目目录的系统。

对于你现在的目标来说，可以把它理解为：

- **当前仓库**：源码仓库，适合二次开发；
- **安装目录中的脚本副本**：给命令行直接调用；
- **用户目录中的模型配置**：给 Claude Code 启动时读取。

所以它确实会“复制一些东西”，但复制的是 **启动脚本和配置文件**，不是从当前代码看就会把整个项目完整再复制出一份。

## 2. 项目结构分析

当前仓库结构非常小，核心文件只有几个：

- `README.md`：项目说明、安装方式、使用方式；
- `cc`：主 Bash 启动脚本；
- `ccs`：与 `cc` 功能相同的 Bash 启动脚本；
- `cc.cmd`：Windows 下的入口包装脚本；
- `ccs.cmd`：Windows 下的入口包装脚本；
- `install.sh`：Mac/Linux 安装脚本；
- `install.bat`：Windows 安装脚本。

其中真正的核心逻辑主要集中在：

1. **`cc` / `ccs`**：负责扫描模型配置、交互选择、添加配置、删除配置、启动 Claude；
2. **`install.sh` / `install.bat`**：负责把脚本复制到用户 PATH 下，并复制模型模板到 `~/.claude/models`；
3. **`cc.cmd` / `ccs.cmd`**：在 Windows 下负责先找到 Git Bash，再转去执行 Bash 版脚本。

从当前版本来看，`cc` 和 `ccs` 两个文件内容一致，`cc.cmd` 和 `ccs.cmd` 也一致。也就是说，项目为了同时支持 `cc` 和 `ccs` 这两个命令，实际上采用了“双文件并存”的方式，而不是只维护一个入口。

## 3. 运行原理

### 3.1 启动入口

在 Windows 下，你通常是执行 `cc.cmd` 或 `ccs.cmd`。这两个脚本先尝试查找 Git Bash，然后去执行同名的 Bash 脚本：

- `cc.cmd:4-6` 查找 `bash.exe`
- `cc.cmd:16` 组装目标 Bash 脚本路径
- `cc.cmd:30` 调用 Bash 执行实际脚本

如果当前目录下找不到 `cc` / `ccs`，它还会回退到：

- `C:\Users\Fansun\.local\bin\cc`
- `C:\Users\Fansun\.local\bin\ccs`

这就是你看到 `C:\Users\Fansun\.local\bin` 很关键的原因。这个目录不是完整项目副本目录，而是 **安装后的命令落地点**。

### 3.2 模型配置来源

主脚本 `cc` / `ccs` 会把配置目录定位到：

- `~/.claude/models`

对应代码位置：

- `cc:20`
- `ccs:20`

之后脚本会扫描这个目录下所有 `*.json` 文件，把文件名当作可选模型别名，把 `ANTHROPIC_MODEL` 读取出来作为菜单描述：

- `cc:25-39`

因此项目本身并不把模型列表写死在代码里，而是 **扫描配置目录动态生成菜单**。

### 3.3 启动 Claude Code

真正启动 Claude Code 的关键逻辑在：

- `cc:168-169`
- `ccs:168-169`

也就是执行：

```bash
claude --settings <模型配置文件>
```

README 里也明确说明了这一点：

- `README.md:181-190`

这说明当前版本的设计重点是：

- 每个模型有自己的独立配置文件；
- 每个 Claude Code 实例用 `--settings` 指向不同配置；
- 从而实现多窗口并行、互不干扰。

这也是当前版本相较旧版本最关键的变化之一。README 明确写了旧版本曾经是“替换 `settings.json`”，现在已经改成 `--settings` 参数方案。

## 4. “复制出另一份”的问题分析

这是你最关心的部分。根据当前仓库代码，项目里确实存在“复制”操作，但性质要区分清楚。

### 4.1 会复制什么

#### Windows 安装脚本会复制：

`install.bat` 会把当前项目中的几个文件复制到用户目录：

- `install.bat:19` 安装目录为 `%USERPROFILE%\.local\bin`
- `install.bat:50` 复制 `cc`
- `install.bat:56` 复制 `cc.cmd`
- `install.bat:63-64` 复制 `ccs` 和 `ccs.cmd`

此外它还会复制模型配置文件到：

- `%USERPROFILE%\.claude\models`

相关位置：

- `install.bat:70-71`
- `install.bat:75-97`

#### Mac/Linux 安装脚本也会复制：

- `install.sh:54` 复制 `cc`
- `install.sh:57` 创建 `ccs` 链接
- `install.sh:63` 创建 `~/.claude/models`
- `install.sh:78` / `install.sh:84` 复制模型配置

### 4.2 不像会复制什么

从当前仓库没有看到以下行为：

1. 没有把整个仓库目录递归复制到另一个位置；
2. 没有自动执行 `git clone` 的逻辑；
3. 没有检测当前仓库再生成一份镜像副本的逻辑；
4. 没有后台守护程序持续同步当前项目目录。

我还专门搜索了 `copy`、`cp`、`clone`、`install`、`settings` 等关键词。结论是：

- 项目中的复制行为集中在 **安装脚本**；
- 复制对象集中在 **启动脚本和配置文件**；
- `.git/logs` 中出现的 `clone` 只是你当初克隆仓库时 Git 自己记录的日志，不是项目代码在运行时又执行了一次克隆。

### 4.3 为什么你会感觉自己电脑里有 3 份

这个感知是合理的，通常会来自下面三类位置：

1. **你最初手动下载/克隆的源码目录**；
2. **安装脚本复制到 `C:\Users\Fansun\.local\bin` 的命令脚本**；
3. **你现在正在看的这个开发目录**。

如果你之前又手动复制过目录，或者从别的路径再次下载过，那么视觉上就会更像“项目自动又复制了一份”。但单从当前项目代码看，它最明确、最稳定会生成的额外文件，是 `.local/bin` 和 `.claude/models` 下的内容，而不是完整仓库副本。

## 5. 当前版本存在的一个重要细节

当前 `cc` / `ccs` 脚本里仍然保留了一个 `prepare_config()` 函数：

- `cc:65-85`
- `ccs:65-85`

这个函数会：

1. 备份 `~/.claude/settings.json`；
2. 把某个模型配置复制到 `~/.claude/settings.json`。

但在当前主流程里，真正启动时使用的是：

```bash
claude --settings "$model_config"
```

也就是说，**现在真正生效的是 `--settings` 启动方案**，而 `prepare_config()` 更像是旧实现遗留下来的逻辑，当前主流程中没有看到它被调用。

这对二次开发很重要，因为它意味着：

- 你需要区分“历史遗留逻辑”和“当前真实运行路径”；
- 如果后续要扩展功能，应优先围绕 `launch_claude()` 这条主路径改，而不是围绕旧的 `settings.json` 覆盖方案改。

## 6. 二次开发应该从哪里入手

如果你的目标是“在不破坏现有使用方式的情况下继续扩展”，建议从下面几个层次理解。

### 6.1 第一层：入口与安装

你先把项目看成一个 CLI 工具：

- 安装层：`install.bat` / `install.sh`
- Windows 入口层：`cc.cmd` / `ccs.cmd`
- 核心逻辑层：`cc` / `ccs`

如果你要改“安装后会不会复制、复制到哪里、安装哪些文件”，重点看：

- `install.bat`
- `install.sh`

如果你要改“命令如何分发到 Bash”，重点看：

- `cc.cmd`
- `ccs.cmd`

如果你要改“模型菜单、启动方式、参数传递、配置管理”，重点看：

- `cc`
- `ccs`

### 6.2 第二层：推荐优先改哪里的代码

从维护成本角度，最值得优先改的是 `cc`。因为它是主逻辑文件，几乎所有核心行为都在这里。

推荐的二次开发方向包括：

1. 给启动过程加更清晰的日志；
2. 增加“调试模式”，打印当前读取的配置路径、Claude 可执行文件路径；
3. 增加“dry-run 模式”，只显示将要执行的命令而不真正启动；
4. 优化模型列表排序与展示；
5. 把 `cc` 与 `ccs` 合并为单一源码，再通过安装脚本生成别名，减少重复维护。

如果你的目标是验证“是否又复制了文件”，最适合改的其实也是安装脚本和启动脚本，因为复制行为只可能发生在这两个阶段。

### 6.3 第三层：哪些地方要特别小心

1. **API Key 写入**：`add_model()` 会把用户输入直接写入 json 文件，属于敏感配置；
2. **Windows/Git Bash 路径兼容**：`USERPROFILE`、`cygpath`、`bash.exe` 查找逻辑不能随便动；
3. **命令名兼容**：当前同时支持 `cc` 与 `ccs`，修改时要注意两个入口不能失配；
4. **不要误用旧逻辑**：`prepare_config()` 看起来像现成方案，但当前启动主路径已不依赖它。

## 7. 如何验证它是否会再次复制

这是你接下来二次开发前最值得做的验证。

### 7.1 先明确“验证对象”

你需要验证的不是“Claude 会不会启动”，而是：

1. 运行 `install.bat` 时，会新增或覆盖哪些文件；
2. 运行 `cc` / `ccs` 时，会不会把项目再复制到别的目录；
3. 运行 `cc add` / `cc remove` / `cc reset` 时，会修改哪些配置文件。

### 7.2 当前代码推导出的预期结果

#### 执行安装脚本时
预期会变化的路径：

- `C:\Users\Fansun\.local\bin\cc`
- `C:\Users\Fansun\.local\bin\cc.cmd`
- `C:\Users\Fansun\.local\bin\ccs`
- `C:\Users\Fansun\.local\bin\ccs.cmd`
- `C:\Users\Fansun\.claude\models\*.json`
- 用户 PATH 环境变量（Windows）

#### 执行普通启动命令时
例如：

```bash
cc kimi
```

当前代码下预期行为是：

- 读取 `~/.claude/models/kimi.json`
- 调用 `claude --settings <该文件>`
- **不应复制整个项目目录**

#### 执行配置管理命令时
- `cc add`：新增一个 json 配置文件；
- `cc remove xxx`：删除一个 json 配置文件；
- `cc reset`：删除 `~/.claude/models` 下全部 json 文件。

### 7.3 最推荐的验证方法

建议你后面真的做实验时，按下面顺序验证：

#### 方法 A：对目录做前后对比
在执行前后，分别观察：

- `C:\Users\Fansun\.local\bin`
- `C:\Users\Fansun\.claude\models`
- 当前项目目录的同级目录
- 你怀疑曾被复制到的那个目录

重点看：

- 是否新增完整项目目录；
- 是否只是新增 `cc/ccs` 脚本；
- 是否只是新增 json 配置。

#### 方法 B：给安装脚本和启动脚本加日志
如果你准备二次开发，最直接的办法是：

- 在安装脚本里打印每一次复制目标；
- 在 `cc` 启动前打印当前实际读取的配置文件路径；
- 必要时打印当前 Bash 脚本真实路径。

这样你就能非常直观看到，到底是“脚本被安装到用户目录”，还是“整个项目被复制”。

#### 方法 C：专门做一次最小验证
只运行下面两类命令：

1. 安装命令；
2. 启动命令。

分开验证，不要混在一起。这样一旦看到新增文件，就能判断它到底是安装步骤造成的，还是启动步骤造成的。

## 8. 对你当前二次开发的建议

如果你现在的目标是先“搞清楚项目再下手改”，建议按下面顺序推进：

- `README.md`

如果是从“方便以后维护”的角度出发，我最建议优先做的是：

## 9. 结论

这个项目本质上是一个 **Claude Code 多模型命令行启动器**。它的主要职责是：

- 管理模型配置；
- 提供 `cc` / `ccs` 入口；
- 通过 `claude --settings` 使用不同模型配置启动 Claude Code。

从当前仓库代码分析结果看：

1. **它会复制文件，但主要是安装脚本复制启动脚本和配置文件；**
2. **没有看到会自动把整个项目仓库再复制一份的明确逻辑；**
3. **你提到的 `C:\Users\Fansun\.local\bin` 很可能就是安装后脚本落地的位置，而不是完整项目副本；**
4. **如果你要进行二次开发，重点应放在 `cc`、`install.bat`、`cc.cmd` 这三类文件；**
5. **如果你要验证“会不会再次复制”，最好的做法是对安装前后、启动前后做路径级对比，并给关键脚本加日志。**
